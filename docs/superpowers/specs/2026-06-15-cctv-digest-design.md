# CCTV_Digest 设计规格

- 日期：2026-06-15
- 状态：已通过设计评审，待写实现计划
- 作者：Esther + Claude（brainstorming）

---

## 0. 一句话本质

一台**私人信号读取仪**：把新闻联播这份"宣传"还原成*带时间维度、带置信度的投资假设*（政策主题 → 行业 → 示例标的），并始终区分【内容】/【信号】/【推断】。

---

## 1. 背景与第一性原理

新闻联播不是"新闻"，而是中央每日 19:00 发布的**经过精心编排的信号**。真正有价值的信息不在"事实层"，而在**编辑选择层**：

- **上场顺序与时长** —— 议题前排/给更多分钟 = 权重上升。
- **措辞/提法变化（提法学）** —— 新提法出现、旧表述消失、关键词微调，常领先于实际政策。
- **出场领导人与语境**。
- **连续报道的"敲鼓"（drumbeat）** —— 持续强调 → 政策/运动 → 预算与资源配置 → 行业景气。这是"宣传 → 投资假设"的因果链。

新闻联播有稳定的三段式结构（领导动态 / 发展成就·民生 / 国际）。单日是噪声，**真正的脉络在时间序列里**。

### 用户的 5 个真实工作（Jobs-to-be-done）

1. 今天该关注什么？（快 + 信号）
2. 风向在变吗？（变化检测 —— 最硬的 alpha）
3. 长期主线是什么？（定位）
4. 这条线索靠谱吗？（验证 / 可审计）
5. 它对我的组合意味着什么？（可执行）

**关键洞断**：这些能力共用同一套引擎；差异化最高的是 Job②（变化）与 Job③（长期定位）。产品重心据此设定：**首页 = 河流（定位）；日切片 = 雷达（变化）+ 晨报（解读）的融合**。

---

## 2. 产品定位与范围

- **定位**：深度优先的投研工具。解读可直接、UI 够清楚即可、合规口径宽松。
- **访问控制**：**内容页面（河流脉络 / 每日解读 / 探索 / 主线）公开，无需登录，所有人可访问**；**系统管理（模型与供应商配置等）需鉴权**。详见 §15。
- **范围内（v1）**：每日抓取与解读、全量历史回填、河流脉络、融合日切片、探索信息图、主线详情、系统管理（模型/供应商配置 + 鉴权）。
- **Phase 2（范围外）**：探索·"问联播" RAG 对话；品质打磨。
- **非目标**：不做实时荐股、不替代用户的投资判断。

---

## 3. 核心引擎（与运行环境无关的 IP）

四级流水线：

1. **抓取 Ingest** —— 每日 + 全量回填。原文存 Blob，元数据入库。详见 §5。
2. **廉价结构化提取 Extract**（便宜模型，跑**全部历史**）—— 见 §4.1。喂时间序列。
3. **变化检测 / 雷达 Radar**（确定性计算为主 + 轻量 LLM）—— 见 §4.2。
4. **深度解读 + 脉络合成**（强模型）—— 见 §4.3、§4.4。

提取与回填是一次性/批量的廉价工作；深度解读只跑"新每日 + 按需回溯"；脉络合成定期或按需重算。

## 4. 解读框架（核心 IP）

### 4.1 廉价结构化提取（全历史）

逐条目产出：

- 三段式分段：领导动态 / 发展成就·民生 / 国际。
- 播出顺序（rundown position）。
- 时长代理：字数 / 相对位置（无视频时长时的代理量）。
- 出场领导人/机构（实体）。
- 关键词与提法（归一化后的规范词 + 别名）。
- 涉及行业 + 极性（利好 / 利空 / 中性）+ 强度。
- 条目摘要（1–2 句）。

日级汇总：三段式统计（条数/时长占比）、当日提法清单（含首现标记）。

### 4.2 变化检测 / 雷达（vs 滚动 90 日基线）

- `新提法首现`：归一化提法在历史中首次/近期罕见地出现。
- `口径翻转`：同一话题的措辞语义级转向（如"防风险"→"促进平稳健康发展"）。**需 LLM 判定**，确定性方法无法捕捉。
- `敲鼓升温/降温`：提法/行业频率相对基线的显著 delta。
- `排序跃迁`：议题在 rundown 中的位置显著前移/后退。

每个雷达事件带类型、对象、量级，并连接到对应的当日解读或长期轨迹。

### 4.3 深度解读（强模型，新每日 + 按需回溯）

- 当日 **Top-3 信号**。
- 每条信号的**三层读**，逐层可展开、各带置信度：
  1. 政策主题（常驻）
  2. 受影响行业 / 板块
  3. 示例标的（**高不确定性层，单独隔离，标注"线索·非建议"**；优先板块/ETF 层面）
- 每条信号关联到所属**主线**，并标注是否"来自雷达"（变化驱动）。

强制三分层叙事：**【内容】说了什么 →【信号】为什么是它/为什么是现在/为什么这样排 →【推断】对政策/行业/资金意味着什么**。

### 4.4 脉络合成（强模型 + 客观聚类，定期/按需）

**主线数量不预设**，由数据客观决定。

- **成立门槛**（一条主线必须同时满足，避免过度拟合）：
  - *共现聚类*：一组相关提法/行业在语料中稳定共现。
  - *持续强度*：在某窗口内的强调强度高于基线。
  - *跨时间持续性*：跨越足够长的时间窗，而非一次性事件。
- **生命周期**：主线可涌现 / 合并 / 分裂 / 消退；每次重算输出当前客观存在的主线集合（可能 4 条，也可能 12 条）。
- 每条主线含：名称、轨迹（时间序列强度）、当下解读、支撑证据（哪些天/条目在敲鼓）。
- **反过拟合原则**：宁可少一条，不可为凑数或凑"好看的数量"而造主线；门槛是客观的，不是目标数量。

## 5. 数据来源（已于 2026-06-15 验证）

- **主源**：`cn.govopendata.com/xinwenlianbo/YYYYMMDD/` —— 干净的按日期 URL，存档可回溯至约 1985 年（2009+ 完整覆盖）。**有反爬**（裸请求返回 HTTP 403）→ 抓取必须带真实请求头 + 限速/退避。
- **兜底（2017→今，结构化）**：Tushare `cctv_news` API。
- **兜底（2023→今，免费 Markdown，GitHub Actions 每日自动更新）**：`github.com/DuckBurnIncense/xin-wen-lian-bo`（`news/YYYYMMDD.md`）。
- 其他：`mrxwlb.com`（每日文字）、`cnopendata.com`（商业数据集）。

回填以 govopendata 为主覆盖 pre-2017；缺日用兜底源补。来源构成兜底链（主源失败 → 次源）。

## 6. 技术架构（方案 A · Vercel + 免费档优先）

- **前端/应用**：Next.js（App Router）on Vercel。
- **数据库**：Neon Postgres —— 承载时间序列与聚合查询。**早期用 Neon 免费档**（原文存 Blob，库内只存精简后的结构化字段以省空间）。
- **对象存储**：Vercel Blob —— 原文文字稿（早期用免费额度）。
- **LLM**：经**多供应商抽象层**（见 §16）调用，**默认 DeepSeek V4**（Pro 跑深读/脉络、Flash 跑提取/雷达），**具体供应商与模型可在系统管理中配置**。供应商至少支持：Vercel AI Gateway、DeepSeek 官方 API、硅基流动（SiliconFlow），以及其他 OpenAI 兼容端点。
- **调度**：Vercel Cron（每日自动跑流水线；Hobby 档支持每日一次的 cron，足够）。
- **回填**：一次性批处理脚本，**在本地/CI 跑**并直接写 Neon（避免 serverless 超时，也不占 Vercel 额度）。
- **鉴权**：仅保护系统管理（见 §15）；内容页面公开。
- **基建成本**：早期尽力全免费档（Vercel Hobby + Neon free + Blob 免费额度），仅 LLM 按量付费（DeepSeek 每月约 $1 级）。增长后再按需升级，见 §17。

## 7. 数据模型（关键实体）

- `broadcast_day`：date、原文 Blob 引用、三段式统计、来源、状态。
- `item`：day、order、segment、length_proxy、text、summary。
- `tifa`（提法）：规范词、first_seen、aliases[]。
- `tifa_mention`：day、tifa、count、context。
- `sector_signal`：day、sector、polarity、strength。
- `daily_interpretation`：day、top_signals[]（每条含三层 + 各层置信度 + 关联 thread + from_radar 标记）。
- `radar_event`：day、type（new_tifa / flip / drumbeat_up / drumbeat_down / order_jump）、target、magnitude、detail。
- `thread`：id、name、color、status（active/merged/split/faded）、生命周期元数据。
- `thread_point`：thread、period、intensity（轨迹）。
- `thread_evidence`：thread、day、item。
- `app_config`：键值配置 —— 各流水线环节（extraction / radar / deep / thread）当前选用的 **供应商 + 模型** 及参数（effort/温度等）；由系统管理编辑。**不存 API Key**（密钥放环境变量，见 §15/§16）。
- `pipeline_run`：每次流水线运行记录 —— 日期、各环节所用供应商/模型、token 用量、成本估算、状态/错误（便于审计与成本追踪）。

> 模型需支持 4.4 的主线生命周期（合并/分裂用 status + 关系记录表达）。

## 8. 页面、信息架构与交互

### 视觉语言
- **River**（暗色电影感流图）当骨架/导航主轴。
- **大字编辑刊**（Editorial：超大字号、强网格、巨大编号、克制留白）当阅读页。
- 暗色基调统一；热色强调（口径翻转/高置信）。
- 系统管理走简洁后台样式即可，不必套用河流美学。

### 页面
1. **首页 = 全屏 River**：发展主线流图，宽度 = 被强调力度；鼠标扫时间读"横截面"读数；**点某天 → 动效（河流压缩成顶部 context 条，每日解读上浮淡入，~0.75s 缓动）进入每日解读**；可"返回长河"。River 需渲染**可变数量**主线，并对长尾小主线做归并/阈值 + 下钻。
2. **每日解读（融合日切片）**：顶部小河流定位 → §雷达（今日变化，连接到解读/轨迹）→ §晨报（Top-3 信号 + 三层可展开解读）→ §三段式编排。叙事链：河流定位 → 雷达发现变化 → 晨报给出解读。
3. **探索 Explore**：关键词/提法时间序列、行业敲鼓热力矩阵（行业×时间）、提法首现时间线。纯信息图。
4. **主线详情**：单条主线的编辑刊深度页（轨迹 + 当下解读 + 证据日）。
5. **系统管理（`/admin`，需鉴权）**：模型与供应商配置（每个流水线环节选 供应商+模型+参数）、流水线运行/成本查看、手动触发补跑。内容页面公开，仅此页受保护。
6. **Phase 2**：探索·问联播（RAG 对话）。

## 9. 自动化与成本

每日 Cron（Vercel Hobby 每日一次）：抓取 → 提取 → 雷达 → 深度解读 →（按需触发脉络重算）。

- 模型按系统管理的配置选取（默认 DeepSeek V4：Flash 提取/雷达、Pro 深读/脉络）。
- 回填一次性、在本地/CI 跑；每日增量在 Cron 跑。
- 任务**幂等**；缺日可补跑；失败可重入；每次运行写 `pipeline_run` 记录 token/成本。
- 成本详见 §17（早期免费基建 + DeepSeek，月成本约 $1 级）。

---

## 10. 数据质量与错误处理

- 来源**兜底链**（主源 403/缺失 → 次源）。
- 缺日处理与补跑。
- 原文长度/完整性校验（异常短/空 → 标记并兜底）。
- LLM 输出 **schema 校验 + 失败重试**。
- 产出带置信度与"待确认"标记；区分仪式性噪声与真实信号。

---

## 11. 可信度 / 认知谦逊（产品内建）

- 置信度徽章（高/中/低）贯穿每层解读。
- "单日 ≠ 政策落地"等提示；口径翻转默认标"需二次确认"。
- 高不确定性的**标的层单独隔离**并标注"线索·非建议"。
- 三层结构强制区分【内容】/【信号】/【推断】。
- 脉络遵循反过拟合原则（§4.4）。

---

## 12. 测试策略

- 提取解析器单元测试。
- 抓取解析 **golden-file** 测试（锁定各来源 HTML/MD 解析）。
- 时间序列聚合**快照测试**。
- 解读质量：一小批人工标注日做 **eval**（信号召回/误报、行业映射合理性）。
- 关键页面视觉核对（River 渲染、动效、每日解读）。

## 13. 分期 / 构建顺序

- **P0** 脚手架：Next.js+Vercel（Hobby 免费档）、Neon free、Blob、**LLM 多供应商抽象层（§16，默认 DeepSeek V4）**、建表。
- **P1** 抓取 + 回填（本地批处理，含 403 应对与兜底链）。
- **P2** 全量廉价提取（DeepSeek Flash）→ 时间序列。
- **P3** 全屏 River 首页（真实数据）+ 探索基础。
- **P4** 每日深度解读（DeepSeek Pro）+ 雷达 + 融合每日解读页。
- **P5** 脉络合成（客观聚类）+ 主线详情页。
- **P6** 每日 Cron 端到端自动化。
- **P7** 系统管理：鉴权（§15）+ 模型/供应商配置 UI（§16）+ 运行/成本查看。
- **P8（后续）** 品质打磨 + 问联播 RAG。

## 14. 风险与开放问题

- **抓取稳定性**：govopendata 反爬强度可能变化；兜底链是关键缓解。
- **解读质量与幻觉**：靠 schema 校验、置信度分层、标的层隔离、eval 集缓解。
- **脉络客观性**：聚类门槛需调参，防止过拟合或漏报；首版后用历史回测校准。
- **成本**：全量回填总量需在 P2 前用小样本估算确认。
- **模型档位**：默认 DeepSeek V4（Pro/Flash 分工），但**可在系统管理切换供应商与模型**（见 §16）；具体在 P2/P4 用真实文字稿 `count_tokens` 校准。
- **Neon 免费档容量**：6,400 天结构化数据可能逼近免费档上限；缓解：原文存 Blob、库内只存精简字段、必要时升级或归档。
- **Hobby 商用条款**：Vercel Hobby 为非商用；纯个人使用可接受，若转商用需升级 Pro。

---

## 15. 鉴权与访问控制

- **内容页面公开**：河流脉络 / 每日解读 / 探索 / 主线，**无需登录，所有人可访问**。
- **仅系统管理（`/admin`）需鉴权**。早期用**最简单、零成本**方案：单一管理员密钥（环境变量）+ 签名会话 Cookie，由 Next.js middleware 保护 `/admin` 与相关写接口。无第三方依赖、免费。
- **未来可升级**：如需多用户/更强鉴权，再接 Clerk（Marketplace 免费档）或 Auth.js。
- **密钥安全**：各 LLM 供应商的 API Key 存 Vercel 环境变量/密钥，**不入库、不在前端暴露、不在管理 UI 显示**；管理 UI 只切换"用哪个供应商+模型"。

## 16. 模型与供应商配置

- **多供应商抽象层**：流水线通过统一接口取得"当前环节该用的模型"，底层适配多家供应商：
  - **Vercel AI Gateway**（`provider/model` 字符串）
  - **DeepSeek 官方 API**（OpenAI 兼容端点）
  - **硅基流动 SiliconFlow**（OpenAI 兼容端点）
  - 其他 **OpenAI 兼容** 端点（可扩展）
  - DeepSeek 与 SiliconFlow 均为 OpenAI 兼容，可用同一 OpenAI-compatible 客户端 + 各自 `baseURL`/`apiKey` 统一接入。
- **按环节配置**：extraction / radar / deep / thread 四个环节各自可选 {供应商, 模型, 参数}，存于 `app_config`，在系统管理 UI 编辑；默认 DeepSeek V4（Flash 提取/雷达、Pro 深读/脉络）。
- **密钥**：每个供应商一个环境变量（如 `DEEPSEEK_API_KEY`、`SILICONFLOW_API_KEY`、`AI_GATEWAY_API_KEY`）。UI 选供应商，运行时按所选供应商读取对应密钥。
- **可观测**：每次运行把所用供应商/模型、token、成本写入 `pipeline_run`，在系统管理可查。

## 17. 成本估算（DeepSeek V4 + 免费基建）

单价（每百万 tokens，第三方聚合站列表价，落地前以官方为准）：DeepSeek V4 **Flash $0.09/$0.18**、**Pro $0.435/$0.87**（缓存命中输入更低）。Token 假设：提取 ~12k in/3k out，深读 ~15k in/8k out，回填 ~6,400 天。

- **一次性回填**（Flash 提取全历史）≈ **$10–15**。
- **每月 LLM**：提取/雷达（Flash）≈ $0.07 + 深读（Pro，每日）≈ $0.40 + 脉络（Pro，周级）≈ $0.09 ≈ **$0.6/月**（含按需深挖也在 $1–2/月）。
- **基建**：早期 Vercel Hobby + Neon free + Blob 免费额度 = **$0**。
- **早期月成本 ≈ $1 级（几乎只有 LLM）**；一次性 ≈ $12。
- **省钱杠杆**：解读框架做成稳定前缀走缓存（DeepSeek 缓存输入极低）、Cron 排在 DeepSeek 错峰时段、回填用 Flash。
- **不确定性**：token 比例与 thinking 输出待用 `count_tokens` 实测；即便偏差 2–3 倍仍在 $1–3/月量级。
